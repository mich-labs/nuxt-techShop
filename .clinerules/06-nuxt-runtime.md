# Nuxt runtime / SSR-safety

- НИКОГДА не использовать `process.env` напрямую в коде, который может выполняться и на клиенте (composables, mappers, `shared/lib`, компоненты). `process.env` доступен только на сервере — на клиенте тихо вернёт `undefined`; баг проявится не при первой SSR-загрузке, а только после client-side навигации между страницами, что делает его сложным для обнаружения.
- Публичные переменные окружения (URL CMS, публичные хосты и т.п.) — только через `runtimeConfig.public`, объявленный в `nuxt.config.ts`. Секретные переменные — через `runtimeConfig` без `.public`, доступны только на сервере.
- `useRuntimeConfig()`, `useNuxtApp()` и другие Nuxt-composables вызывать синхронно в setup-фазе компонента/composable или в самом начале функции — до `await`, если `experimental.asyncContext` не гарантированно включён в проекте. Не прятать `useRuntimeConfig()` внутри вложенных helper-функций (`getMediaUrl` и т.п.), вызываемых из мапперов после `await` внутри `useAsyncData` — это падает с ошибкой "composable called outside setup". Вместо этого: доставать значение (`cmsUrl`) один раз в начале composable страницы, передавать параметром явно во все мапперы.
- Любой код, трогающий `window`/`document`, оборачивать в `onMounted`/`onUnmounted` (никогда не на верхнем уровне `<script setup>`) — на сервере этих объектов не существует. Обязательно снимать листенеры в `onUnmounted` во избежание утечек памяти при частой SPA-навигации.
- Не использовать `document.querySelector` для доступа к своим же элементам шаблона — использовать `ref` на элемент шаблона (`const el = ref<HTMLElement | null>(null)` + `ref="el"` в template).
- Простые вещи вроде отслеживания скролла/resize — писать через собственный composable-обёртку (например `useWindowScroll`) с `onMounted`/`onUnmounted` внутри, не через стороннюю библиотеку по умолчанию (см. `03-stack.md`) и не разбрасывая `addEventListener` по компонентам напрямую.
