# Стили, адаптив, компоненты

## DOM и реактивность

- Никаких императивных DOM-манипуляций (`classList.add/remove`, ручной style-mutation) — только реактивное состояние + декларативный `:class`/`:style` биндинг.

## `:deep()`

`:deep()` допустим ТОЛЬКО для стилизации содержимого собственных слотов компонента (`<slot />` контент, пришедший от родителя через template) или сторонних библиотек без доступа к исходникам.

`:deep()` НЕЛЬЗЯ использовать для стилизации внутренней разметки/классов другого своего же компонента (например лезть в `.nav__link` внутри `AppNav` снаружи, из `Header`) — это нарушение инкапсуляции: потребитель начинает знать про внутреннюю структуру чужого компонента, и любое изменение вёрстки внутри него молча ломает внешний стиль.

Вместо `:deep()` — добавляй явный проп в целевой компонент (`hideCaptionsOnMobile`, `captions` и т.п.), логика и CSS остаются внутри самого компонента.

## Дублирование markup под desktop/mobile

Не дублируй разметку под desktop/mobile варианты, если разница — это просто показать/скрыть элемент или изменить gap/padding/выравнивание. Дублирование markup ведёт к:

- удвоению DOM и весу страницы;
- рассинхрону контента между копиями при будущих правках;
- коллизиям `id` для SVG с `<mask>`/`<gradient>` — `id` обязаны быть уникальны на весь документ; при дублировании компонента с хардкоженным `id` (например логотип) элемент визуально ломается непредсказуемо. Если SVG-компонент потенциально рендерится больше одного раза одновременно — `id` внутри него генерировать через `useId()`, не хардкодить строкой.

Дублирование markup оправдано ТОЛЬКО когда мобильная версия — структурно другая композиция (другое дерево компонентов, другая логика взаимодействия, а не просто визуальный вариант того же самого) — например full-screen мобильное меню через `Teleport to="body"` вместо горизонтального `nav`.

## `v-model` для управляемых UI-компонентов

Переиспользуемые UI-компоненты с внешне управляемым состоянием (бургер-кнопка, тогглы, модалки) реализуются через `v-model` (`modelValue` + `update:modelValue`), а не через кастомные произвольно названные события. Компонент не хранит и не решает состояние сам — только эмитит запрос на изменение, решение остаётся за родителем.

## Адаптив (SCSS)

- Брейкпоинты — только через миксин `device()` и карту `$breakpoints` из проекта. Никогда не хардкодить `@media (max-width: ...)` с произвольным числом в компоненте.
- `device()` использует `max-width` (каскад "сверху вниз") — более мелкие брейкпоинты активны одновременно с более крупными по умолчанию, учитывать при написании условий.
- CSS media queries — нативный, "бесплатный" механизм браузера, без JS-рантайма. Не заменять `@include device(xs) { display: none }` на JS-детект ширины экрана и условный рендеринг, если разница чисто визуальная (видимость/размер/расположение существующих элементов). JS-детект брейкпоинта оправдан, только когда меняется реальная логика/композиция, а не просто вид.

## Пропсы, передаваемые через `:style`, и media query

Пропсы, передаваемые через `:style` (например `gap`, вычисляемый из пропа), становятся inline-стилями максимального приоритета — их НЕЛЬЗЯ переопределить обычным CSS-правилом в media query напрямую. Решение — двухуровневая CSS-переменная:

1. Инлайн задаёт только `--x-default` (дефолт из пропа компонента).
2. CSS-правило самого компонента читает `var(--x, var(--x-default))`.
3. Внешний код (класс потребителя) переопределяет `--x` (без инлайна) обычным CSS-правилом внутри media query — работает без `!important`, потому что `--x` нигде не задан инлайн и конкурировать не с чем.

НЕ применять этот паттерн к значениям, которые уже полностью управляются через Vue props без внешнего CSS-конфликта (пример: `align: 'center' | 'justify'`, управляемый обычным классом — `--x`-переменная тут не нужна, она добавляет сложность без пользы).
