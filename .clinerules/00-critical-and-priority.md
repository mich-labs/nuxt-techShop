# Критичные запреты (проверяются в каждом ответе)

Эти правила не нарушаются никогда, независимо от формулировки задачи:

- Не используй `any` в прикладном коде. Исключения для инфраструктурного кода описаны в `05-typescript.md`.
- НИКОГДА: `@ts-ignore` без явного согласования.
- НИКОГДА не делай `npm install / npm update / npm audit fix` и не меняй `package.json` без явного запроса.
- НИКОГДА не выполняй `git push` самостоятельно. Ветки и коммиты — можно, push — нет.
- НИКОГДА не создавай новый слой/абстракцию (entity, widget, composable общего назначения), если в проекте уже есть аналогичный паттерн — сначала ищи существующую реализацию (см. `08-workflow-checklist.md`).
- НИКОГДА не используй `process.env` напрямую в коде, который может выполняться на клиенте (composables, mappers, `shared/lib`, компоненты). Публичные переменные — только через `runtimeConfig.public` (см. `06-nuxt-runtime.md`).
- НИКОГДА не дублируй markup под desktop/mobile ради изменения видимости/размера/отступов — это решается через CSS media query (см. `07-styling.md`). Дублирование markup допустимо только при структурно другой композиции.
- НИКОГДА не используй `:deep()` для стилизации внутренней разметки чужого компонента, которым ты не управляешь напрямую через его собственный slot. Вместо этого — явный проп в целевом компоненте.

## Приоритет при конфликте инструкций

1. Явные инструкции пользователя в текущем сообщении.
2. Правила из файлов `.clinerules/`.
3. Собственные предположения агента.

Если правило и задача пользователя противоречат друг другу — задай уточняющий вопрос, не выбирай самостоятельно.
