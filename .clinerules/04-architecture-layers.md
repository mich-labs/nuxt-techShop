# Архитектура: FSD-lite поверх Nuxt

Проект использует адаптацию Feature-Sliced Design под файловый роутинг Nuxt. Полного канонического FSD нет и не нужно — ниже описаны реальные правила проекта.

## Слои (сверху вниз)

`app/pages` (нативный Nuxt-роутинг) → `widgets` → `entities` → `shared`.

Правило импорта: слой не может импортировать вышестоящий слой или другой слайс своего же уровня.

- `entities` импортирует только `shared`. Не импортирует другую `entity` или `widget`.
- `widgets` импортирует `entities` и `shared`. Не импортирует другой `widget` напрямую (композиция виджетов происходит на уровне `pages`).
- `shared` не импортирует ничего из `entities`/`widgets` — это низлежащий, безусловный слой (UI-примитивы, лишённые домена: `AppLink`, `AppBanner`, `AppStack` и т.п., плюс `shared/lib` — переиспользуемые чистые хелперы вроде `mapLink`, `getMediaUrl`).

## `app/pages/*.vue`

Максимально тонкие: только композиция виджетов + вызов composable страницы (`useHomePage`, `useProductPage`) + `useSeoMeta`. Никакой бизнес-логики, мапперов, `useAsyncData`/fetch-вызовов напрямую в файле страницы — всё это выносится в composable страницы.

Composable для каждой уникальной страницы — свой (`useHomePage.ts`, `useAboutPage.ts`). Не создавай единый generic-хук с параметрами "url + mapper" — это преждевременная абстракция. Обобщение (`useEntityBySlug<TDto, TModel>`) допустимо только когда есть 2-3 реально одинаковых типовых случая с идентичной механикой (например разные типовые детальные страницы с одним и тем же паттерном "получить сущность по slug + один и тот же тип мапинга").

## Критерий: entity vs часть widget

Заводи отдельный `entities/*`, если:

- Домен переиспользуется в ≥2 разных, независимых контекстах (не "в теории может понадобиться", а реально используется).
- В Strapi это отдельный переиспользуемый компонент/репитер (`banner (5)` repeatable), а не разовое фиксированное поле формы.

Оставляй модель локальной внутри `widgets/*`, если:

- Модель имеет смысл только в контексте одной конкретной композиции/вёрстки (например "секция с двумя картинками под конкретный layout").
- В Strapi это фиксированное одиночное поле (`bannerSection1`), не список.

Не выноси в `entities` "на всякий случай" — сначала используй локально, выноси при появлении второго реального потребителя (YAGNI). Общий переиспользуемый `shared/ui` компонент (например `AppBanner`, используемый и в widget, и потенциально в entity) — НЕ признак того, что модели данных должны быть объединены в одну сущность.

## Структура сегментов слайса

Сегменты: `ui/`, `model/`, `api/`, `lib/`, `index.ts`.

- Создавай папку сегмента, только если в нём больше одного файла. Если сегмент — это ровно один файл (например только `types.ts`), клади его в корень слайса без обёртки в папку `model/`. Не создавай пустую вложенность заранее.
- Entity может не иметь `ui/` вообще, если у неё нет собственного целостного визуального представления (данные используются, но раскладываются по-разному каждый раз внутри чужой разметки потребителя).
- `index.ts` — единственная точка входа СНАРУЖИ слайса. Внутри слайса файлы импортируют друг друга напрямую по относительному пути, не через собственный `index.ts`.

### Что экспортировать через `index.ts`, а что нет

Экспортировать: Model-типы (`export type { BannerModel }`), функции-мапперы (`export { mapBanner }`).

НЕ экспортировать (приватные детали реализации): DTO-типы (сырая форма ответа Strapi/API — не должны покидать файл, где определены), внутренние helper-функции, не предназначенные для внешнего использования.

### Исключение для `.vue`-компонентов

Nuxt авто-импортирует все `.vue` файлы из `components/` глобально по имени файла независимо от `index.ts` (если в `nuxt.config.ts` явно не отключена авто-регистрация для конкретного пути через `components: [{ path, global: false }]`). По умолчанию в проекте авто-регистрация не отключена — не утверждай, что `index.ts` защищает доступ к `.vue` компонентам. `index.ts` пиши для TS-логики слайса (мапперы, типы, локальные composables), не для самих `.vue` файлов.

## DTO / Model / Mapper

- **DTO** — сырая форма ответа API, берётся из сгенерированного typed-клиента Strapi (`import type { XxxGetPayload } from '~~/cms'`), никогда не пишется руками с нуля.
- **Model** — нормализованная форма для UI-слоя, то, что видит компонент.
- Между DTO и Model всегда есть явная функция-маппер. Не инлайнить DTO-generic прямо в сигнатуру функции — вынести именованный тип `type XxxDto = ...GetPayload<{...}>` рядом с Model в `types.ts`.
- Не вводить термин "Record" как отдельное понятие наравне с DTO/Model без явного собственного БД-слоя на фронте — двух терминов достаточно.
- Повторяющиеся мелкие структуры (например `{ url, isExternal }` → `AppLinkProps`) выносить в `shared/lib` как отдельный переиспользуемый маппер (`mapLink`), а не копипастить в каждом мапере отдельно. Модель в разных местах должна ссылаться на один и тот же тип, а не заново описывать ту же форму.
