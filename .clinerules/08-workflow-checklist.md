# Перед написанием кода

1. Изучи связанные файлы и слой, к которому относится изменение (`entities`/`widgets`/`shared`/`pages`) — см. `04-architecture-layers.md`.
2. Изучи существующие реализации аналогичных сущностей/виджетов (мапперы, model-типы, composables страниц).
3. Определи существующий подход в проекте: если для похожей задачи уже есть паттерн (например `useLinkMapper`, `getMediaUrl`, `device()`-миксин) — используй его, не изобретай параллельный.
4. Представь план изменений в формате из `01-role-and-communication.md`.
5. Не начинай писать код, пока не понял архитектуру конкретного участка и не определил слой.
6. Не создавай новую архитектуру/абстракцию, если аналогичная уже существует.

# Если информации недостаточно

Не выдумывай:

- Не придумывай существующие компоненты, composables, миксины.
- Не придумывай существующие классы или методы.
- Не придумывай форму DTO Strapi — сверяйся с реальным сгенерированным типом из `~~/cms` или спрашивай пользователя.

Если чего-то не хватает — задай вопрос, либо изучи соседние файлы проекта.

# Работа с терминалом

Не запускай `npm install`, `npm update`, `npm audit fix` без явного запроса.

Не изменяй `package.json` без необходимости.

# Игнорируемые директории

Не просматривай `node_modules`, `.nuxt`, `.output`, `dist`, `coverage`, `.git`, если пользователь явно этого не попросил.

# Стиль кода

- Следуй существующему стилю проекта.
- Не меняй стиль оформления существующего файла при правке несвязанного участка.
- Не переименовывай сущности без необходимости.
- Не выполняй рефакторинг файлов, если это не требуется задачей.
- Изменяй только те участки кода, которые относятся к задаче.
